# pyviz — Python Codebase Visualizer

> A hybrid CLI + web tool that ingests a Python codebase and emits a navigable
> call/dependency graph an engineer onboarding to that codebase can actually trust.

**Live viewer:** https://pyviz-visualizer.vercel.app  
**Sample output:** FastAPI repo graph — see `sample-output/` in the repo

## 1. Form factor decision

**Hybrid**: a deterministic Python analysis core (CLI) that emits `graph.json`,
consumed by a Next.js + React Flow web viewer.

**Why local Python for the analysis core:**
- Re-export resolution, MRO computation, and scope-chain name binding are
  Python-language problems. The `ast` module and `importlib` machinery give
  direct access without reimplementing Python import semantics in another language.
- Runtime tracing (`sys.monitoring` / `sys.setprofile`) *requires* being in-process
  with the target interpreter. A non-Python host cannot install profile hooks on a
  test suite it didn't run.
- A CLI run over a checked-out repo is hermetic and re-runnable. No network
  round-trips, no shared-state backend to debug. The artifact is diffable in CI.

**Why a separate web viewer (not a static HTML blob):**
- Navigation of a 200+ file repo is a UX problem. It needs filtering, search,
  progressive disclosure, and provenance toggles — a hand-rolled HTML file can't
  do ego-graph expansion or server-side dagre layout.
- Clean handoff contract: the CLI's only obligation to the viewer is
  "produce `graph.json` conforming to the schema in §5." The two halves are
  independently testable.

**The handoff:**
```
pyviz analyze ./your-repo  →  ./pyviz-out/graph.json
# Drop graph.json in the browser → fully navigable graph
```

## 2. Local setup — CLI

**Requirements:** Python ≥ 3.10

```bash
git clone https://github.com/<your-handle>/pyviz-visualizer
cd pyviz-visualizer
pip install -e .           # installs the `pyviz` command

# Analyze a local repo (static analysis only)
pyviz analyze ./path/to/your-repo --out ./pyviz-out

# With runtime tracing (requires the repo's own test suite)
pyviz analyze ./path/to/your-repo \
    --trace \
    --test-command "pytest -q" \
    --out ./pyviz-out

# Verify §5.3 invariant on a produced graph
pyviz verify ./pyviz-out/graph.json

# Structural diff between two graph snapshots (stretch feature)
pyviz diff ./pyviz-out-v1/graph.json ./pyviz-out-v2/graph.json
```

### Full CLI reference

```
pyviz analyze <path>
  --source-root PATH         override auto-detected source root
  --out DIR                  output directory (default: pyviz-out/)
  --trace / --no-trace       run runtime tracing via test suite (default: off)
  --test-command TEXT        command used to drive tests (default: "pytest -q")
  --tracer auto|monitoring|setprofile|settrace   backend (default: auto)
  --include GLOB             include only matching paths
  --exclude GLOB             exclude matching paths

pyviz verify <graph.json>   check §5.3 and dangling-edge invariants (CI-safe exit codes)
pyviz diff <a.json> <b.json> structural diff between two graph artifacts
```

**Exit codes:** `0` = clean; `1` = §5.3 invariant violation (imports edge targets
`__init__.py` for a symbol defined elsewhere). Non-zero exit is CI-friendly.

### Running the test suite

```bash
pip install -e ".[dev]"
pytest -v
# → 36 tests, covering P3 unit tests + full-pipeline integration over all fixtures
```

## 3. Local setup — Viewer

**Requirements:** Node.js ≥ 18

```bash
cd viewer
npm install --legacy-peer-deps
npm run dev        # → http://localhost:3000
```

**Production build:**
```bash
npm run build && npm start
```

**Deploying to Vercel:**
1. Push the repo to GitHub.
2. Go to [vercel.com/new](https://vercel.com/new), import the repo.
3. Set **Root Directory** to `viewer/`.
4. Vercel auto-detects Next.js; use the default build settings.
5. Click **Deploy**. No env vars required for local-file mode.

Once deployed, drop any `graph.json` generated by the CLI into the browser.

## 5. graph.json schema overview

The artifact is the contract between the CLI and the viewer. Schema version: `1.0`.

### Node

```jsonc
{
  "id": "a1b2c3d4e5f60718",        // sha1("{kind}::{relpath}::{qualname}")[:16]
  "kind": "module|package|class|function|method|coroutine|variable|external",
  "name": "FastAPI",
  "qualname": "FastAPI",            // CPython-style: "Outer.method", "f.<locals>.g"
  "module": "fastapi.applications",
  "file_path": "fastapi/applications.py",   // POSIX-normalized, relative to repo root
  "line_start": 48,
  "line_end": 512,
  "attributes": {
    "is_async": false,
    "method_kind": null,            // "instance" | "classmethod" | "staticmethod" | null
    "is_abstract": false,
    "framework_entrypoint": false,  // true for @app.get routes, @celery.task, etc.
    "decorators": ["dataclass"],
    "mro": ["fastapi.applications.FastAPI", "starlette.applications.Starlette", "object"],
    "all_is_dynamic": false
  },
  "caveats": []                     // human-readable strings shown as badges in the UI
}
```

### Edge

```jsonc
{
  "id": "f0e1d2c3b4a59687",         // sha1("{source}::{target}::{kind}")[:16]
  "source": "<node_id>",
  "target": "<node_id>",
  "kind": "imports|calls|instantiates|inherits|decorates|defines",
  "provenance": "static|runtime|both",
  "confidence": "resolved|heuristic|dynamic-unresolved",
  "evidence": {
    "static":  { "file": "app/main.py", "line": 22 },
    "runtime": { "call_count": 1428 }
  },
  "caveats": []
}
```

### Top-level graph document

```jsonc
{
  "schema_version": "1.0",
  "repo": { "path": "...", "commit": "abc123", "source_roots": ["fastapi"] },
  "nodes": [...],
  "edges": [...],
  "stats": {
    "n_nodes": 1840,
    "n_modules": 312,
    "n_classes": 180,
    "n_functions": 920,
    "n_edges": 4100,
    "n_calls_static": 3200,
    "n_calls_runtime": 620,
    "n_calls_both": 280,
    "n_init_targeted_definition_edges": 0   // must be 0 — §5.3 invariant
  },
  "global_caveats": ["runtime edges bounded by test suite coverage"]
}
```

### Node identity

`node_id = sha1(f"{kind}::{relpath}::{qualname}")[:16]`

- `relpath` is POSIX-normalized (forward slashes, relative to repo root) for
  machine-independent, reproducible IDs.
- Line numbers are *metadata*, not identity — `ast.lineno` and `co_firstlineno`
  can differ subtly after decorators, which would break the static↔runtime join.
- Two pipeline runs over the same commit produce byte-identical `graph.json`
  (nodes and edges sorted by `id` before emit).

## 6. Design decisions

### Why `sys.monitoring` over `sys.settrace`

`sys.settrace` fires a callback on **every line** — paying per-line cost for a
call graph is pure overhead. `sys.monitoring` (PEP 669, Python 3.12+) exposes
`PY_START` events at call granularity with minimal overhead. For Python < 3.12,
`sys.setprofile` is the correct classical tool (call-granular, no per-line cost).
`settrace` is documented as a fallback only.

### The re-export resolver (P3) — the flagship pass

An AST-only tool draws `from fastapi import FastAPI` as an edge to
`fastapi/__init__.py`. That edge is wrong: `FastAPI` is *defined* in
`fastapi/applications.py`. The init file merely re-exports it.

P3 resolves this transitively. Algorithm: memoized, cycle-safe recursion —
(1) defined here → done, (2) explicit binding → follow it, (3) wildcard source
→ search `__all__`, (4) dead end → unresolved. Every (module, name) pair is
resolved exactly once. The headline invariant (`n_init_targeted_definition_edges
== 0`) is automatically checked on every CI run and in `pyviz verify`.

### Relative import resolution (P2)

`from .core import X` in `pkg/__init__.py` must resolve to `pkg.core`, not `core`.
The key insight (following CPython's `importlib._bootstrap._resolve_name`): for
`__init__.py` files, the module name *is* the package — do not strip the last
dotted component. For regular modules (`pkg/core.py`), strip the last component
to get the containing package. P2 detects this from the file path suffix.

### Soundness over recall

When the static analyzer cannot prove a callee (attribute call on unknown type,
`getattr(obj, name)()`, higher-order callbacks), it emits **nothing** — no false
edge, no guess. A `dynamic-unresolved` placeholder is stored as metadata at the
call site but never rendered as a proven edge. The runtime tracer recovers these
as `provenance: runtime` edges.

### Module-first progressive disclosure

The default view renders only module/package nodes (~dozens, not thousands).
Clicking a module node expands its class/function/method children. This is the
design decision that keeps the viewer usable on 200+ file repos — React Flow with
2000+ nodes simultaneously is unusable.

## 7. What this tool handles well

| Feature | How |
|---|---|
| `__init__.py` re-export resolution | P3 transitive resolver; tested with fixtures + FastAPI spot-checks |
| `__all__` and wildcard imports | P3 respects `__all__` when statically known; marks `all_is_dynamic` when not |
| Import ≠ call distinction | `imports` and `calls` are separate, independently filterable edge kinds |
| Decorators (plain, factory, framework-registration) | P4 `decorates` edges; `@app.get` routes tagged `framework_entrypoint` |
| Inheritance + abstract bases | `inherits` edges; ABC detection; `is_abstract` flag |
| Async surfaces | `is_async` flag on all coroutine nodes; `coroutine` kind |
| Static ↔ runtime provenance | Every call edge labeled `static` / `runtime` / `both` |
| Framework entrypoints | Route handlers invisible to static analysis appear as `runtime`-only after tracing |
| Cycle safety | Re-export cycles return `status: cycle`, surface a caveat, never loop |
| Deterministic output | Two runs on same commit → byte-identical `graph.json` |

## 8. Consciously deferred scope

These are explicit, defensible decisions — not oversights.

| Deferred item | Rationale |
|---|---|
| Full type inference for attribute calls | Undecidable in general. `obj.method()` where `obj`'s type is not locally provable is marked `dynamic-unresolved` and recovered at runtime. Speculative points-to analysis would add complexity without correctness guarantees. |
| C extensions / Cython | Opaque `external` nodes; no internal edges. Separate tooling (e.g., `pycparser`) required. |
| Cross-language analysis | Python only. Out of scope. |
| Hosted multi-user persistence | Supabase + Google OAuth is specified but deferred. Local-first ships; hosted mode is a thin additive layer (one `graphs` table keyed by user id). Adding it for local use would be theater. |
| Dynamic `__all__` enumeration | Detected and flagged as `all_is_dynamic`. Never guessed — a wrong wildcard expansion is worse than a caveat. |
| Speculative MRO for dynamic bases | When a base class is not statically resolvable, MRO is marked `partial` and dependent call edges are downgraded to `heuristic`. |
| `<locals>` markers in qualnames | P2 currently uses `Outer.method.nested` rather than CPython's full `Outer.method.<locals>.nested`. Affects deep-nesting runtime join precision; primary use cases (module → class → method) are unaffected. |
| Type-flow analysis via mypy/pyright output | Stretch feature; would improve attribute-call resolution significantly but requires a separate tool pass. |
| Circular import detection | Detectable from the `imports` edge graph; deferred as a stretch feature. |
| Repo diff between two commits | `pyviz diff` is stubbed; structural diff logic is correct but the viewer doesn't render it yet. |

## Project structure

```
pyviz-visualizer/
├── pyviz/                    # Python analysis core (install with pip install -e .)
│   ├── cli.py                # Click CLI: pyviz analyze | verify | diff
│   ├── models.py             # Shared immutable data structures (all passes)
│   └── pipeline/
│       ├── p1_discovery.py   # Module discovery → ModuleMap
│       ├── p2_parse.py       # AST parsing → SymbolTables (relative import resolution)
│       ├── p3_resolve.py     # Re-export resolver (transitive, memoized, cycle-safe)
│       ├── p4_edges.py       # Static edge extraction (imports, defines, inherits, decorates, calls)
│       ├── p5_trace.py       # Runtime tracing (sys.monitoring / sys.setprofile)
│       ├── p6_merge.py       # Merge static + runtime → provenance labelling
│       └── p7_emit.py        # Emit graph.json + run-manifest.json
├── viewer/                   # Next.js 15 + React Flow web viewer
│   ├── app/page.tsx          # Landing page: file upload + full graph UI
│   ├── components/
│   │   ├── GraphView.tsx     # React Flow graph with progressive disclosure
│   │   ├── NodePanel.tsx     # Selected node side panel (neighbors, caveats, attributes)
│   │   └── FilterBar.tsx     # Kind / edge / provenance / confidence filter chips
│   └── lib/
│       ├── types.ts          # TypeScript types mirroring graph.json schema
│       └── layout.ts         # Dagre server-side layout computation
├── fixtures/                 # Hand-built packages for invariant testing (§12.1)
│   ├── reexport_chain/       # from .core import X — resolves to core, not __init__
│   ├── wildcard/             # from .core import * — respects __all__
│   ├── cycle/                # re-export cycle — must terminate
│   ├── decorators/           # plain / factory / framework decorator edges
│   ├── getattr_dispatch/     # dynamic-unresolved call sites
│   └── dynamic_all/         # __all__ = base + [...] — flagged all_is_dynamic
└── tests/
    ├── test_p3_resolve.py    # P3 unit tests with hand-built SymbolTables
    └── test_invariants.py    # Full-pipeline integration + §12.4 automated invariants
```
